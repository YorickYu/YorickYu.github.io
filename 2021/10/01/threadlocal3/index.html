ThreadLocal 缺点及改进方案

发表于 2021-10-01 更新于 2022-06-20 分类于 java

FastThreadLocal

问题一、 hash冲突，线性探测，效率低。
问题二、内存泄漏，在get，set方法中会主动清理无用的entry，职责不明，效率低。
优化方案：
1 可以优化成链表+红黑树
2 给每个threadlocal加一个唯一标志 static final int
这样每个threadlocal就都有一个自己的位置了，且因为final不可改变。同时原本threadlocalmap的结构由弱引用的Entry数组改变成了一个Object数组（Entry里面弱引用了ThreadLocal作为Key，放弃了原本map的形式，改成使用唯一标识index作为下标，这样就避免了冲突）

数组第一个位置放置了所有threadlocal对象的set，用于清除操作，之所以Object数组其他位置可以直接存value，是因为每个threadlocal都有一个对应的index唯一下标
PS：因为是通过下标的方式存储，所以Object数组在扩容时是根据index，且不需要进行rehash，提升了速度

3 结合上面Object数组的改进。核心思路是避免了内存泄漏。由于数组的存放时间复杂度都是O(1)，所以这个问题迎刃而解
