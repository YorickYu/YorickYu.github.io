秒杀服务设计

发表于 2021-09-02 更新于 2022-06-20

关键词

单一职责、库存预热、反扒、流量削峰、限流和降级

说明

1 单一职责
因为秒杀一般面对的高并发场景，在秒杀活动前后的流量非常大，平时的流量很少。一般建议将秒杀服务独立部署，平时只部署少量的机器维持C端展示，在活动开始前按情况动态扩容

2 库存预热
通常在秒杀活动开始前，会将活动商品的库存写入redis，可以根据提供库存的量级来设计redis的存放规则。引入分段锁的概念

3 反扒
一般秒杀服务都会接入反扒避免恶意攻击，如果公司没有反扒团队，也可以用其他手段代替：比如活动真实链接需要在活动前1s才下发，或者自定义规则对秒杀接口进行加密等

4 流量削峰
如果你们公司的秒杀活动的QPS已经超过单机redis承受的极限，那么要考虑做流量削峰。最简单的机制是增加用户参与的流程：比如下单前输入验证码等（验证码的形式是很多样的）

5 限流和降级
讨论限流和降级时，说明秒杀活动的QPS已经非常高了，需要通过限流和降级优先保证服务稳定性。最好的方法是在网关层做限流，如果没有条件可以做本地限流，触发限流后返回一个友好的“抢光了”页面

其实秒杀服务很适合用MQ进行削峰处理，主要是按业务场景来做技术选型

1 使用数据库事务保证redis可回滚

2 定时任务增量，或者DTS实时消费数据

3 秒杀无超卖

前言

服务间进行RPC调用时难免会遇到header传递问题

通常推荐的做法是：

1 声明一个 RequestInterceptor 子类，然后实现apply 方法

2 在 apply 方法中获取 HttpServletRequest request = ((ServletRequestAttributes) RequestContextHolder.getRequestAttributes()).getRequest();

3 通过 request.getHeader(key); 方法获取想要的param值，组装成Map

4 最后调用 requestTemplate.headers(Map<String, Collection<String>> headers); 传递

问题

根据 RequestContextHolder.getRequestAttributes() 源码中可看出，请求参数是存在 ThreadLocal 中的，这就导致了在异步调用时，子线程中获取 RequestContextHolder.getRequestAttributes() 会发生空指针异常。

这个问题也可以归结为 异步调用时数据传递问题 或 多线程间数据共享问题

解决

@RequestHeader

这种最简单的方法往往可以最直接的解决问题。

可以看到在 exampleHeader 方法中声明了一个 RequestHeader 注解用来传递参数

这样你就可以通过参数传递的方式在主线程中声明 HttpHeaders 对象，然后把这个对象当做参数传递到子线程中使用

这种方法使用比较灵活，但是也有缺点：header中需要传递的参数很难约束。因为只是需要一个 HttpHeaders 对象，那么到底对象中应该传递哪些参数？key的命名如何？这些问题都要通过后续的方法来约束。

《Effective-java》 归纳整理

发表于 2021-07-11 更新于 2022-06-20

使用枚举类型强化单例模式

单元素的枚举类型已经成为实现Singleton的最佳方法

建造者模式

多个构造器参数可变时考虑建造者模式

Lombok中@Builder注解支持

避免创建不必要的对象

1 String类型合理利用元空间，不要创建String实例 （JVM）

2 不可变类尽量使用静态工厂方法而非构造器。例如 Boolean.valueOf() 优于 new Boolean()

3 评估是否有必要使用延迟初始化，延迟初始化常见在创建单例时使用，有静态内部类和DCL两种方式 （DCL单例）

4 避免在循环中误用自动拆箱装箱

5 如果是非常重量级的对象，可以考虑维护自己的对象池来避免对象频繁创建 （数据库连接池）

避免内存泄漏

1 正确编写代码，可以查阅一些网上常见的内存泄漏案例
例如：HashMap中使用自定义类型时要实现hashCode和equals方法，这是个老生常谈的问题了

2 使用缓存时要合理定义缓存的粒度，最重要是不论使用什么缓存架构，都要设置过期时间和过期策略（缓存架构概念篇）

3 对于监听器或流的使用，主要利用try finally进行显示关闭

数据迁移 SOP（不停机、分片）

发表于 2021-07-06 更新于 2022-06-20 分类于 数据持久化 ， SOP

双写开启

insert： 新老库同时写入
update：
1 新库已存在这条数据 -> 更新新老库数据
2 新库不存在这条数据 -> 写入新库并更新老库

双写提前开启也是为了保证迁移失败时数据可回滚

全量数据迁移 daemon

记录任务开始和结束的 timestamp(start ~ end)

读老库，写新库
1 新库已存在这条数据 -> 判断老数据 gmt_modified > start timestamp -> 更新新库
1 新库不存在这条数据 -> 写入新库

ps: 如果写入失败要日志记录相应的id，后续做增量追加或重试

增量数据迁移 xxljob

循环校验新老库的数据，如果存在不一致，将数据同步:

UPDATE 新表名称 SET 列名称 = 新值 WHERE id = id AND gmt_modified < end timestamp

这个定时任务可以持续几天，对数据校验及修正

切流

将读取数据流量切换到新库

双写关闭，完整迁移

其他事项

1 binlog读取 — canal

增量数据迁移常见的解决方式为 MySQL binlog读取。可以借助运维工具 canal

2 分片后的主键

通常设计一个大表时，可以预留分表可用主键，因为需要保证单次查询的原子性，但是分表后又很难兼顾复杂情况的查询。

所以可以在设计表时预先根据表中高频查询的字段建立索引规则。然后将这个规则融合在主键中。

如果没有预先埋好分表策略的主键，可以临时做一个mapping表做检索，也可以解决分表后的复杂查询问题
ps: 通常这种索引表可以支撑亿级数据

3 服务拆分

将迁移服务独立拆分，避免以后重复造轮子。（迁移方案可以使用策略模式，兼容不同的情况）

4 使用游标

使用游标的方式从旧库分批select数据，经过rehash后批量插入（batch insert）到新库，此处需要配置jdbc连接串参数rewriteBatchedStatements=true才能使批处理操作生效。
