翻车实例：云上mongoDB升级导致节点连接超时

发表于 2021-10-13 更新于 2022-06-20

背景

服务从原本的自建mongo集群拆分至云上mongo主从节点（1主2从），迁移后发现云上mongo有不定时抖动的情况导致接口RT升高。
随后第三方人员定位到是云上mongo中有一个节点存在慢日志毛刺，分析后定位到该机器磁盘有抖动问题，故建议做节点迁移。

操作流程大致是新节点上线并将新老节点均设置为隐藏，新老替换成功后将老节点下线，新节点放开。
理论上操作对业务服务是无感的，仅有服务闪断重连的风险。

问题

新节点上线，老节点隐藏后Java服务正常，PHP服务连接超时，服务瘫痪

原因定位

驱动流程：
1 mongoDB链接数据库，驱动会根据isMaster()返回的hosts中的主从ip节点进行连接。这里的返回的ip默认是物理机地址。
2 第三方云服务mongoDB修改了内核，保证isMaster()返回的hosts中的ip是云上环境可访问的VIP地址。
异常原因：
PHP驱动不但会访问isMaster()返回的hosts中的节点，还会探测passive中的节点，而passive节点中的ip没有经过处理，默认返回的是物理机地址，与CVM不通导致连接超时

PS：
1 hosts中的节点降低优先级为0时会出现在passive中
2 探测passive节点是PHP驱动的独有行为

方案改进

1 更换hosts节点时，老节点立即下线，不要让其出现在passive中
2 …

gossip 协议

发表于 2021-10-09 更新于 2022-06-20

流行病算法

基于六度分隔理论 — 一个人通过6个中间人可以认识世界上的任何人。

特点

信息的传播速度非常迅速，网络交互次数不会很多。

协议执行过程

1 设定种子节点的扩散周期（1秒）
2 每个节点随机选择与他相连的n个节点进行扇出（fanout设定为6个）
3 收到消息的节点不反馈，不回传（A->B，B不反馈结果，也不会再发给A）

缺点

1 消息最终是通过多个轮次的散播而到达全节点的，因此它必然会存在各节点状态不一致的情况。无法准确预算单条消息传播全节点的时间
2 随机扇出，不可避免消息重复发送至相同节点，有一定网络传输冗余

传播模式

1 反熵：以消除各节点差异为目的。每次节点两两交换自己的所有数据，目标是整个网络各节点完全的一致。通信负担非常大
2 传谣：以传播消息为目的。仅发送新数据，目标是节点数据增量同步。通信负担较小。

一般来说，为了在通信代价和可靠性之间取得折中，需要将这两种方法结合使用。

应用

分布式系统中达成”最终一致性”的协议
1 比特币网络中信息同步
2 Redis sentinel 集群中信息同步

常见数据模型对比

发表于 2021-10-07 更新于 2022-06-20 分类于 数据持久化

文档模型、关系模型、图状模型

区别

1 文档模型也叫“层次模型”，没有严格的数据模式，可以将任意的键值对添加到文档中（很容易表示数据的层次，即1:n的关系），在读取时无法保证文档中包含哪些字段，更准确的术语是“读时模式”

2 关系模型是“写时模式”，创建时定义好一系列规则，写入时必须遵循。可以表示简单的多对多关系

“读时模式”类似于动态（运行时）类型检查，“写时模式”类似于静态（编译时）类型检查

3 随着数据之间的关联越来越复杂，图模型会更有优势。

图模型由两部分组成：顶点和边。前者表示实体、后者表示关系。
使用场景：社交网络、web、导航系统等

关联

他们在各自的目标领域中都足够优秀。也有各自的优势

文档模型和图模型有一个相似特点，就是他们都是“读时模式”，不会对存储的数据强加校验，应用过程中可以更好的适用需求的变化

关系模型的优点在于，可以笨拙的模拟出文档和图模型。所以相对来说适用范围更广

PS：文档型灵活、数据局部性具有性能优势，使用方便。关系型对复杂关系的表达更简洁。可以根据不同的业务场景选择或结合使用，毕竟数据模型是相互补充的

数据查询语言

数据查询语言通常分为 声明式 和 命令式 两种。

以关系模型中的SQL举例，它就是声明式的语言。而文档和图状模型中的IMS和CODAYSL都是命令式的。

命令式语言通常会详细写出查询执行的整个过程，逐行遍历直到查到想要的结果
声明式语言则只需要写出需要的数据和满足什么条件即可
两者的区别有点类似于白盒和黑盒

关系型数据库，例如MySQL为声明式语言提供了很多自动优化的空间、使用简单。并且，声明式语言更适合并行执行

声明式语言更适合并行执行?
因为命令式语言通常有严格的查询逻辑，很难在多核（线程）或多台机器（进程）上并行执行。

日志

发表于 2021-10-03 更新于 2022-06-20 分类于 java

基本日志规范

1 包含 方法名、入参、出参
1 避免重复打无意义的日志
1 生产环境不要打debug日志，有条件的话不同级别的日志分文件（或者存储kafka）
1 核心功能的日志尽量打完整
2 不能直接使用日志系统（Log4j、Logback）中的 API，而是使用日志框架SLF4J中的API。SLF4J是门面，可以保证不修改代码前提下更换底层日志框架
3 建议使用{}占位符
4 不要使用e.printStackTrace()
e.printStackTrace()打印出的堆栈日志跟业务代码日志是交错混合在一起的，通常排查异常日志不太方便。
e.printStackTrace()语句产生的字符串记录的是堆栈信息，如果信息太长太多，字符串常量池所在的内存块没有空间了，打爆内存

traceid

traceid的概念是非常核心的，从你的服务（甚至一个链路）接收到一个C端的HTTP请求时，它可能会经历一个很长的过程才返回给C端客户。
这个漫长的链路上任何一处错误都可能导致请求的失败，在繁杂的日志文件中精准查找问题是很难的。所以traceid的概念是非常关键的。

如过是比较成熟的架构，这个traceid可以由 网关 生成并从请求头中传递，同样在 RPC框架 中应该也同样支持traceid的传递。如果是自己实现的话，可以参考下面的处理方式。

统一处理方式

1 Annotation & AOP

从 ProccedingJoinPoint 中可以获得比较多的信息，包括入参、方法名、还有注解中传递的参数（这里的可操作性很强）。
这种方式功能性更强，但是缺点也很明显，就是需要我们手动添加，多人维护项目时也比较难统一功能

2 Filter & RequestInterceptor

这种方式的实现基础是每个请求都会经过filter的处理，你可以做一些事的同时并且从请求上下文中获取信息，打印日志。
这种方式的功能比较单一，可以把请求的出参入参输出一下。

核心的思路是生成traceid且尽量避免代码入侵，且遵循上面几个基本日志规范

ThreadLocal 缺点及改进方案

发表于 2021-10-01 更新于 2022-06-20 分类于 java

FastThreadLocal

问题一、 hash冲突，线性探测，效率低。
问题二、内存泄漏，在get，set方法中会主动清理无用的entry，职责不明，效率低。
优化方案：
1 可以优化成链表+红黑树
2 给每个threadlocal加一个唯一标志 static final int
这样每个threadlocal就都有一个自己的位置了，且因为final不可改变。同时原本threadlocalmap的结构由弱引用的Entry数组改变成了一个Object数组（Entry里面弱引用了ThreadLocal作为Key，放弃了原本map的形式，改成使用唯一标识index作为下标，这样就避免了冲突）

数组第一个位置放置了所有threadlocal对象的set，用于清除操作，之所以Object数组其他位置可以直接存value，是因为每个threadlocal都有一个对应的index唯一下标
PS：因为是通过下标的方式存储，所以Object数组在扩容时是根据index，且不需要进行rehash，提升了速度

3 结合上面Object数组的改进。核心思路是避免了内存泄漏。由于数组的存放时间复杂度都是O(1)，所以这个问题迎刃而解

补足理解欠缺

发表于 2021-09-07 更新于 2022-06-20 分类于 稳定性

前言

现在有一个很通用的场景：
数据存储在DB中，读写请求基本八二分布，服务中有一套定时任务去维护 缓存数据。
读请求处理会优先走缓存，写请求会作用在DB中，然后更新缓存。
大概就是这么一个普通的场景，那么下面有几个问题:
1 系统局部失效时，如何保证数据的正确性？
2 系统降级方案如何设计？怎么保证降级时的数据一致性？
3 DB中数据不断扩充，如何对系统进行扩展？
4 如何提升系统代码的可读性？

其实再简单的技术方案也多多少少会有问题，根据不同的业务场景也有多种解决方案，通过技术改造、架构设计、算法辅助来增加系统的完整性的同时，也会增加系统的复杂度，越复杂的系统就越容易出错。
这些问题不是目的，
目的在于根据业务场景设计技术方案时要具备这种思维。考虑服务的 可靠性、扩展性 和 可维护性。

可靠性

即使故障发生时，系统也可以正常运行

故障率：机器故障、软件故障、人为错误。

其中避免人为错误的方式：
1 以最小出错的方式来设计系统
2 模块分离
3 充分的测试
4 全面监控及快速故障响应

可扩展

量化业务增长，熟悉系统负载，当负载到达极限时如何对系统架构升级

负载：吞吐量、RT、TP99/TP95、排队延迟、长尾效应
垂直扩展 VS. 水平扩展：服务优先考虑水平扩展、数据库优先考虑垂直扩展

可维护

降低系统复杂度，使系统便于修改和适配新场景和新业务

1 运维性：监控、告警
2 简单性：用简单的方式实现
3 可演化：敏捷开发、设计模式

秒杀服务设计

发表于 2021-09-02 更新于 2022-06-20

关键词

单一职责、库存预热、反扒、流量削峰、限流和降级

说明

1 单一职责
因为秒杀一般面对的高并发场景，在秒杀活动前后的流量非常大，平时的流量很少。一般建议将秒杀服务独立部署，平时只部署少量的机器维持C端展示，在活动开始前按情况动态扩容

2 库存预热
通常在秒杀活动开始前，会将活动商品的库存写入redis，可以根据提供库存的量级来设计redis的存放规则。引入分段锁的概念

3 反扒
一般秒杀服务都会接入反扒避免恶意攻击，如果公司没有反扒团队，也可以用其他手段代替：比如活动真实链接需要在活动前1s才下发，或者自定义规则对秒杀接口进行加密等

4 流量削峰
如果你们公司的秒杀活动的QPS已经超过单机redis承受的极限，那么要考虑做流量削峰。最简单的机制是增加用户参与的流程：比如下单前输入验证码等（验证码的形式是很多样的）

5 限流和降级
讨论限流和降级时，说明秒杀活动的QPS已经非常高了，需要通过限流和降级优先保证服务稳定性。最好的方法是在网关层做限流，如果没有条件可以做本地限流，触发限流后返回一个友好的“抢光了”页面
