日志

发表于 2021-10-03 更新于 2022-06-20 分类于 java

基本日志规范

1 包含 方法名、入参、出参
1 避免重复打无意义的日志
1 生产环境不要打debug日志，有条件的话不同级别的日志分文件（或者存储kafka）
1 核心功能的日志尽量打完整
2 不能直接使用日志系统（Log4j、Logback）中的 API，而是使用日志框架SLF4J中的API。SLF4J是门面，可以保证不修改代码前提下更换底层日志框架
3 建议使用{}占位符
4 不要使用e.printStackTrace()
e.printStackTrace()打印出的堆栈日志跟业务代码日志是交错混合在一起的，通常排查异常日志不太方便。
e.printStackTrace()语句产生的字符串记录的是堆栈信息，如果信息太长太多，字符串常量池所在的内存块没有空间了，打爆内存

traceid

traceid的概念是非常核心的，从你的服务（甚至一个链路）接收到一个C端的HTTP请求时，它可能会经历一个很长的过程才返回给C端客户。
这个漫长的链路上任何一处错误都可能导致请求的失败，在繁杂的日志文件中精准查找问题是很难的。所以traceid的概念是非常关键的。

如过是比较成熟的架构，这个traceid可以由 网关 生成并从请求头中传递，同样在 RPC框架 中应该也同样支持traceid的传递。如果是自己实现的话，可以参考下面的处理方式。

统一处理方式

1 Annotation & AOP

从 ProccedingJoinPoint 中可以获得比较多的信息，包括入参、方法名、还有注解中传递的参数（这里的可操作性很强）。
这种方式功能性更强，但是缺点也很明显，就是需要我们手动添加，多人维护项目时也比较难统一功能

2 Filter & RequestInterceptor

这种方式的实现基础是每个请求都会经过filter的处理，你可以做一些事的同时并且从请求上下文中获取信息，打印日志。
这种方式的功能比较单一，可以把请求的出参入参输出一下。

核心的思路是生成traceid且尽量避免代码入侵，且遵循上面几个基本日志规范

ThreadLocal 缺点及改进方案

发表于 2021-10-01 更新于 2022-06-20 分类于 java

FastThreadLocal

问题一、 hash冲突，线性探测，效率低。
问题二、内存泄漏，在get，set方法中会主动清理无用的entry，职责不明，效率低。
优化方案：
1 可以优化成链表+红黑树
2 给每个threadlocal加一个唯一标志 static final int
这样每个threadlocal就都有一个自己的位置了，且因为final不可改变。同时原本threadlocalmap的结构由弱引用的Entry数组改变成了一个Object数组（Entry里面弱引用了ThreadLocal作为Key，放弃了原本map的形式，改成使用唯一标识index作为下标，这样就避免了冲突）

数组第一个位置放置了所有threadlocal对象的set，用于清除操作，之所以Object数组其他位置可以直接存value，是因为每个threadlocal都有一个对应的index唯一下标
PS：因为是通过下标的方式存储，所以Object数组在扩容时是根据index，且不需要进行rehash，提升了速度

3 结合上面Object数组的改进。核心思路是避免了内存泄漏。由于数组的存放时间复杂度都是O(1)，所以这个问题迎刃而解

补足理解欠缺

发表于 2021-09-07 更新于 2022-06-20 分类于 稳定性

前言

现在有一个很通用的场景：
数据存储在DB中，读写请求基本八二分布，服务中有一套定时任务去维护 缓存数据。
读请求处理会优先走缓存，写请求会作用在DB中，然后更新缓存。
大概就是这么一个普通的场景，那么下面有几个问题:
1 系统局部失效时，如何保证数据的正确性？
2 系统降级方案如何设计？怎么保证降级时的数据一致性？
3 DB中数据不断扩充，如何对系统进行扩展？
4 如何提升系统代码的可读性？

其实再简单的技术方案也多多少少会有问题，根据不同的业务场景也有多种解决方案，通过技术改造、架构设计、算法辅助来增加系统的完整性的同时，也会增加系统的复杂度，越复杂的系统就越容易出错。
这些问题不是目的，
目的在于根据业务场景设计技术方案时要具备这种思维。考虑服务的 可靠性、扩展性 和 可维护性。

可靠性

即使故障发生时，系统也可以正常运行

故障率：机器故障、软件故障、人为错误。

其中避免人为错误的方式：
1 以最小出错的方式来设计系统
2 模块分离
3 充分的测试
4 全面监控及快速故障响应

可扩展

量化业务增长，熟悉系统负载，当负载到达极限时如何对系统架构升级

负载：吞吐量、RT、TP99/TP95、排队延迟、长尾效应
垂直扩展 VS. 水平扩展：服务优先考虑水平扩展、数据库优先考虑垂直扩展

可维护

降低系统复杂度，使系统便于修改和适配新场景和新业务

1 运维性：监控、告警
2 简单性：用简单的方式实现
3 可演化：敏捷开发、设计模式

秒杀服务设计

发表于 2021-09-02 更新于 2022-06-20

关键词

单一职责、库存预热、反扒、流量削峰、限流和降级

说明

1 单一职责
因为秒杀一般面对的高并发场景，在秒杀活动前后的流量非常大，平时的流量很少。一般建议将秒杀服务独立部署，平时只部署少量的机器维持C端展示，在活动开始前按情况动态扩容

2 库存预热
通常在秒杀活动开始前，会将活动商品的库存写入redis，可以根据提供库存的量级来设计redis的存放规则。引入分段锁的概念

3 反扒
一般秒杀服务都会接入反扒避免恶意攻击，如果公司没有反扒团队，也可以用其他手段代替：比如活动真实链接需要在活动前1s才下发，或者自定义规则对秒杀接口进行加密等

4 流量削峰
如果你们公司的秒杀活动的QPS已经超过单机redis承受的极限，那么要考虑做流量削峰。最简单的机制是增加用户参与的流程：比如下单前输入验证码等（验证码的形式是很多样的）

5 限流和降级
讨论限流和降级时，说明秒杀活动的QPS已经非常高了，需要通过限流和降级优先保证服务稳定性。最好的方法是在网关层做限流，如果没有条件可以做本地限流，触发限流后返回一个友好的“抢光了”页面

前言

服务间进行RPC调用时难免会遇到header传递问题

通常推荐的做法是：

1 声明一个 RequestInterceptor 子类，然后实现apply 方法

2 在 apply 方法中获取 HttpServletRequest request = ((ServletRequestAttributes) RequestContextHolder.getRequestAttributes()).getRequest();

3 通过 request.getHeader(key); 方法获取想要的param值，组装成Map

4 最后调用 requestTemplate.headers(Map<String, Collection<String>> headers); 传递

问题

根据 RequestContextHolder.getRequestAttributes() 源码中可看出，请求参数是存在 ThreadLocal 中的，这就导致了在异步调用时，子线程中获取 RequestContextHolder.getRequestAttributes() 会发生空指针异常。

这个问题也可以归结为 异步调用时数据传递问题 或 多线程间数据共享问题

解决

@RequestHeader

这种最简单的方法往往可以最直接的解决问题。

这种方法使用比较灵活，但是也有缺点：header中需要传递的参数很难约束。因为只是需要一个 HttpHeaders 对象，那么到底对象中应该传递哪些参数？key的命名如何？这些问题都要通过后续的方法来约束。
